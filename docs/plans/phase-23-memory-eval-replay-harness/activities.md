## Golden Fixture Specification

### Location
Fixture sets live under `test/LeanKernel.Eval.Fixtures/` with subdirectories per scenario class:
```
test/LeanKernel.Eval.Fixtures/
├── retrieval/           # recall@k golden scenarios
├── grounding/           # grounded-answer rate scenarios
├── contradiction/       # contradiction leakage scenarios
├── freshness/           # freshness lag scenarios
└── conflict-flagging/   # false-positive conflict scenarios
```

### Format
Each fixture is a JSON file containing:
```json
{
  "fixtureId": "string (guid)",
  "scenarioClass": "retrieval | grounding | contradiction | freshness | conflict-flagging",
  "description": "human-readable scenario description",
  "inputSnapshot": {
    "ingestedDocuments": ["...staged file references..."],
    "existingClaims": ["...pre-seeded canonical claims..."],
    "queryTurn": "the test query to execute"
  },
  "expectedOutputs": {
    "retrieval": { "topKDocIds": ["..."], "minRecall@5": 0.85 },
    "grounding": { "expectedGroundedAnswer": "...", "minGroundedRate": 0.90 },
    "contradiction": { "expectedConflictFlags": ["..."], "maxLeakage": 0.01 },
    "freshness": { "maxFreshnessLagHours": 24 },
    "conflictFlagging": { "expectedFalsePositives": 0, "maxFalsePositiveRate": 0.05 }
  },
  "createdAt": "ISO-8601",
  "lastRecapturedAt": "ISO-8601"
}
```

### Recapture Process
- Fixtures are recaptured via `dotnet run --project test/LeanKernel.Eval.Fixtures -- --recapture` against a sandboxed DB snapshot.
- Recapture snapshots the current ingestion + claim state, re-runs the query, and overwrites `expectedOutputs` with actual results.
- Recaptured fixtures must be reviewed by owner before merging (manual approval gate in PR).
- Recapture cadence: after any Phase 21/22/17 change, or monthly — whichever is sooner.
- Golden fixture DB snapshots are stored as EF Core SQLite files under `test/LeanKernel.Eval.Fixtures/snapshots/` and regenerated by recapture.
