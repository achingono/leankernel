# Phase 10 Activities

## Step-By-Step Activities
1. Confirm the current scoping: `GBrainMemoryClient` builds `memory/{tenantId}/{userId}/{channelId}/{key}` and `IdentityResolver` creates channel-specific identities; document the exact isolation dimensions used by memory, history, and agent state.
2. Decide the identity model (record in `risk-register.md`): a new `PersonEntity` parent, or promote `UserEntity` to the person with linked channel-identity rows. Define a `ChannelIdentity` record mapping `(tenant, channel, native-identifier) -> personId`.
3. Add entities/migrations for the person and channel-identity mapping; keep tenant as the top isolation boundary.
4. Implement identity resolution changes so a request yields both a `personId` and a `channelId`; new channel identities auto-provision an unlinked person, existing linked identities resolve to the shared person.
5. Implement a verified linking flow: initiate from an authenticated surface, deliver a one-time code to the target channel, and merge the channel identity into the existing person on confirmation. Support unlink. Unify this with Phase 06's channel sender→(tenant, user) binding provisioning — the verified link and the channel credential binding are the same act (a confirmed link establishes the binding), so avoid building two separate linking mechanisms.
6. Change memory scope to person-keyed while **retaining the channel dimension**: update `MemoryScope` to carry `personId` and `channelId`, and update `GBrainMemoryClient` to build `memory/{tenantId}/{personId}/{channelId}/{key}` for writes (always the current channel's own scope).
7. Enforce the Phase 06 policy on reads: inject the channel memory-policy resolution contract, compute the effective readable-channel set for the current channel (directional AND: X readable in C iff X shares to C and C accesses X; C always included), and fan the search across each permitted channel namespace (`memory/{tenantId}/{personId}/{channelId}`), de-duplicating results. With the default `*`/`*` policy this spans all the person's channels (unified memory); narrowed policies scope reads down to partial sharing or self-only isolation.
8. Apply the same policy-driven visibility to the person-scoped preference profile and surface it to context assembly on each channel.
9. Guarantee cross-channel 5W1H consistency. On write, extend related-page lookup, dimension classification (`MemoryDimensionClassifier`), and supersession (`MemoryPageKeyBuilder` `facts/{dim}/{subject-slug}/{factId}`, `SupersededBy`/`Supersedes`, `IsRetired`) to consider same-subject/same-dimension pages across the *mutually visible* channel set (channels visible to each other under directional AND), so an updated fact supersedes/merges the prior fact and the 5W1H fields converge to one canonical current version. Retain the authoring channel as provenance metadata. Resolve conflicts deterministically (newest `EffectiveTimestamp` / explicit supersession wins) and flag genuinely irreconcilable conflicts instead of silently overwriting. Do not merge across channels that are not mutually visible (isolated channels may diverge).
10. Add a reconciliation pass that runs when the sharing policy is enabled or widened between channels: converge already-divergent same-entity facts across the newly-shared set (deterministic supersede/merge, provenance retained, conflicts flagged), dry-run-capable and reversible.
11. Handle asymmetric (one-way) visibility without write-back. When a channel is readable-into but not mutually visible, do not supersede/merge across the boundary; instead resolve same-entity/same-dimension collisions at read/context-assembly time via a non-destructive overlay with deterministic precedence (newest `EffectiveTimestamp` or local-authored wins) and provenance labels. Ensure writes only ever mutate the writer's own channel scope so a change in the shared-into channel never propagates back to the shared-from channel, and that revoking the grant removes the overlay with no residual state.
12. Keep `IdentityIsolationKeyProvider` on the agent-session path unless a separate history/session-sharing design is approved. Limit this phase's scope change to `IPermit`, `MemoryScope`, `MemoryProvider`, `GBrainMemoryClient`, the person/profile resolution path, the memory-policy enforcement wiring, and the cross-channel normalization/reconciliation path.
13. Implement a reversible, dry-run-capable migration that rewrites existing `memory/{tenantId}/{userId}/{channelId}/{key}` keys to `memory/{tenantId}/{personId}/{channelId}/{key}` (retaining channelId), merging duplicates deterministically.
14. Add configuration (person-linking requirements, history-sharing policy, conflict-resolution strategy) and startup validation; consume the memory sharing policy defaults/overrides defined in Phase 06 without redefining them.
15. Add tests: with wildcard default the same person shares memory + preferences across two channels; explicit `Share`/`Access` narrowing isolates or partially shares channels via directional AND; a fact updated in one channel supersedes/reconciles the same-entity 5W1H fact in every mutually shared channel (no divergence) while isolated channels diverge; asymmetric one-way sharing surfaces shared-in facts via non-destructive overlay with deterministic precedence and never writes back to the shared-from channel; conflict flagging is deterministic; unlinked identities isolated; tenant never crossed; migration and reconciliation correctness and reversibility.
16. Update `docs/features/identity-partitioning.md` and `memory-pipeline.md` to document person-keyed, channel-retaining memory, the configurable sharing policy enforcement, cross-channel 5W1H reconciliation, asymmetric overlay handling, and linking.

## Review Focus
- Tenant isolation is never weakened by the person layer.
- With the default policy, memory truly follows the person across channels; with narrowed policy, isolation is honored in both directions (directional AND).
- Reads never cross a channel boundary the policy forbids, and writes never leak into another channel's scope.
- Shared channels never hold divergent 5W1H facts about the same entity: supersession/merge spans the mutually visible set and conflicts resolve deterministically with provenance retained.
- Reconciliation only merges facts that are mutually visible (never surfaces or merges a fact into a channel the policy forbids).
- Unlinked/anonymous identities remain isolated (no accidental merging).
- Linking requires verification and cannot be spoofed to hijack another person's memory.
- Migration and reconciliation are loss-free, deterministic, and reversible, and preserve the channel dimension and provenance.
- Scope-relative keys still passed to `GBrainMemoryClient` (no double-prefixing).
